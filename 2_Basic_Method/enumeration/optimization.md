# space complexity

int        4 字节
long long  8 字节
char       1 字节
double     8 字节
pair<int,int> 大约 8 字节

所以：

int a[1000000];

大概是：

1000000 * 4B = 4MB

还好。

但：

int a[10000][10000];

就是：

10000 * 10000 * 4B = 400MB

这个在很多题里直接爆。

# STL 也有额外开销
vector<int> a;

本质上还是数组，比较省。

但：

map<int,int> mp;
set<int> s;

它们每个元素不是只存一个 int，还要存指针、颜色、树结构信息，所以会比数组大很多。

所以如果 n 很大，比如 1e6，能用 vector / 数组解决，就不要随便上 map / set。

**看到题目数据范围，先估算：**

我存了几个 n？
我存了几个 n*m？
我有没有存所有二元组？
我有没有存所有区间？
我有没有开 map/set 存海量元素？

比如：

n = 1e5
存数组 O(n)：可以
存所有数对 O(n^2)：基本不行

n = 1000
存二维数组 O(n^2)：大概可以

n = 1e5, m = 1e5
存二维数组 O(nm)：绝对不行

**枚举可以用循环“经过”很多情况，但不要轻易把所有情况都存下来。**

# 🚀 枚举优化常见技巧

枚举优化并不是只有单纯的“傻枚举”或“双指针”。更准确地说，枚举优化的核心在于**降维**。

## 0. 总原则
枚举优化本质上就做三件事：
1. **少枚举一些东西**
2. **每次判断更快一点**
3. **不把所有枚举结果都存下来**

> **举个例子：**
> 原来是：枚举所有 $i, j, k$
> 优化后：只枚举 $i$，剩下的用 **查询 / 双指针 / 前缀和 / 哈希** 解决。

---

## 1. 排序 + 单调性
这是最常见的一类优化。如果题目中出现以下特征，**先想：排序以后，答案变化有没有方向？**

* **适用关键词**：最大/最小、接近某个值、两数和/三数和、区间长度、能不能满足某个限制。
* **优化逻辑**：比如找三数最接近目标值，可以**固定一个数，剩下两个数用双指针**，这就是利用了排序后的单调性。

**常见代码形式：**
```cpp
sort(a.begin(), a.end());
```
然后配合：**双指针**、**二分** 或 **贪心**。

---

## 2. 前缀和：快速算区间
如果你枚举的是连续区间，最容易从这里优化。

* **适用关键词**：连续区间、子数组、区间和、一段的总和。
* **优化逻辑**：暴力写法需要三层循环（枚举 $l$ -> 枚举 $r$ -> 从 $l$ 加到 $r$）。但如果提前算好前缀和：
  ```cpp
  prefix[i] = a[1] + a[2] + ... + a[i];
  ```
  那么区间 `[l, r]` 的和可以 $O(1)$ 算出：
  ```cpp
  sum = prefix[r] - prefix[l - 1];
  ```
  复杂度直接从 $O(n^3)$ 降到 $O(n^2)$。

---

## 3. 哈希 / set / map：把“找”变成“快速查询”
* **适用关键词**：是否存在、出现次数、配对、两个数满足某关系、频率统计。
* **优化逻辑**：以“两数之和”为例，暴力枚举 $i$ 和 $j$ 需要 $O(n^2)$。但如果你只枚举一个数 `a[i]`，然后问：“我需要的另一个数是不是出现过？”
  ```cpp
  need = x - a[i];
  ```
  利用 `set` / `map` / `unordered_set` 查询 `need`。复杂度可以变为 $O(n)$ 或 $O(n \log n)$。

---

## 4. 计数代替逐个枚举 (🌟 超级重要)
* **适用关键词**：方案数、有多少对、有多少组、数字范围不大、重复值很多。
* **优化逻辑**：有时候你不需要枚举每个元素，只需要知道**某个值出现了多少次**。
  比如数组 `2 2 2 2 2 3 3 3`。如果你枚举每个位置，很慢；但如果统计出 `2 出现 5 次`，`3 出现 3 次`，很多问题就可以转化成组合计数问题。
  * 选两个 2 的方案数 = $C(5, 2)$
  这类题经常从**枚举位置**变成了**枚举数值**。

---

## 5. 二分答案：不枚举答案，改成判断答案是否可行
* **适用关键词**：最大最小值、最小最大值、最长长度、最短时间、最多能做到多少、至少需要多少。
* **优化逻辑**：如果答案范围很大，不能一个一个试，且**答案具有单调性**（如：长度为 5 能做到，长度为 4 也能做到；长度为 6 做不到，长度为 7 更做不到），就可以二分。
  不要顺序枚举答案，而是问：**如果答案是 `mid`，能不能做到？**

**典型代码结构：**
```cpp
while (l <= r) {
    int mid = l + (r - l) / 2;
    if (check(mid)) {
        // 往一边走 (例如缩小范围: r = mid - 1 或放大范围: l = mid + 1)
    } else {
        // 往另一边走
    }
}
```

---

## 6. 剪枝：发现不可能就提前停
* **适用关键词**：搜索、选择若干个、排列组合、回溯、方案枚举。
* **优化逻辑**：有些枚举树很大，但很多分支明显没必要继续。
  * 当前结果已经不可能更优
  * 当前状态已经非法
  * 剩下元素不够用了
  * 当前和已经超过上限
  * 排序后，后面只会更大/更小
* ⚠️ **注意**：不能凭感觉剪，必须保证被剪掉的情况**真的不可能成为答案**。

---

## 7. 预处理：提前算好，枚举时直接查
如果你在枚举过程中反复算同一个东西，就应该考虑预处理。

* **常见预处理对象**：`prefixMax`, `suffixMax`, `prefixMin`, `suffixMin`，区间和，出现次数。
* **优化逻辑**：比如你需要枚举分界点 $k$，答案是“左边最优 + 右边最优”。如果每次枚举 $k$ 都重新算左右两边就很慢。如果提前算好 `leftBest[k]` 和 `rightBest[k]`，枚举分界点时就是 $O(1)$。

---

## 8. 分治 / 折半枚举 (Meet in the Middle)
* **适用关键词**：从 $n$ 个数中选一些、子集和、方案组合、$n$ 大约在 30~40 之间。
* **优化逻辑**：如果要枚举 $n$ 个东西的所有选择，复杂度是 $2^n$。当 $n = 40$ 时直接超时。
  你可以把它拆成两半：前 20 个枚举所有结果（$2^{20}$），后 20 个枚举所有结果（$2^{20}$），然后再合并。
  这个技巧极其强大，把 $2^{40}$ 变成了大约 $2^{20} + 2^{20}$。

---

## 9. 状态压缩：用二进制表示选择情况
* **适用关键词**：$n$ 很小 ($n \le 20$)、每个元素选或不选、所有子集、排列状态、用过哪些东西。
* **优化逻辑**：用一个整数的二进制位表示状态。比如 `10110` 表示第 1、2、4 个东西被选了。本质还是枚举，但把“选择情况”压成一个整数，代码和状态管理更清晰。

**常见写法：**
```cpp
for (int mask = 0; mask < (1 << n); mask++) {
    // 枚举一种选择方案
}
```

---

## 10. DP / 记忆化：不要重复枚举同一个状态
* **适用关键词**：有很多重复子问题、从当前位置往后选择、问最大/最小/方案数、状态可以被描述出来。
* **优化逻辑**：有些枚举题会反复走到同一个状态（如：当前在第 $i$ 个位置，已用多少资源）。每次重新往后枚举会产生大量重复路径。这时候用**记忆化搜索**或**动态规划 (DP)** 记下来：“这个状态我之前算过了”。

---

## 11. 用数据结构维护当前枚举窗口
* **适用关键词**：滑动窗口、连续区间、当前区间最大/最小值、动态加入/删除元素。
* **优化逻辑**：比如求“连续区间里最高值 - 最低值 $\le K$”。如果暴力枚举区间 `[l, r]` 再扫一遍找极值，会很慢。可以用 `multiset` 或 **单调队列** 来维护当前区间的最大值和最小值。
  * 双指针：负责移动区间
  * 数据结构：负责维护区间信息，避免重算

## 💡 最重要的一句话

> **枚举题优化的核心，不是“不枚举”，而是减少枚举维度、加快判断过程、避免重复计算、避免存下所有情况。**

以后看到一个暴力枚举，先别急着否定它，而是问自己：
**这里哪一层枚举可以被“查询、预处理、单调性、计数、二分、记忆化”替代？**
