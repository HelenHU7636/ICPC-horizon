**“要记录的是动态的战局，而不是动态的规则。”**

**“规则”是在执行具体动作之前的所有预判。不只狭义的if else，还包括for循环等所有在动作执行之前必须发生的检查**

❌ 维护“动态的规则” (Dynamic Rules) = 复杂度失控
这就是你一开始的思路。你试图在每一步，都去精确计算出一个“当前允许填入的元素集合”（必须在、可以在、不能在）。

代价是什么？ 这个集合（规则本身）是高度依赖战局的。战局只要变了一点点（比如前面多了一个 0），你这个“允许集合”就得整个推倒重算。

在代码里，这就意味着你要动态维护极其复杂的数据结构（比如不断增删元素的 std::set），并且处理无穷无尽的边界条件。这就是为什么你的大脑会觉得“乱”，因为你在用战局去干涉规则，这在工程上叫作高耦合 (High Coupling)，**之后的元素合法性由之前的元素影响**。

✅ 维护“动态的战局” (Dynamic States) = 降维打击
这就是让你豁然开朗的那个简单思路，也就是状态机模型 (State Machine)。

规则被剥离出来了： 规则变成了绝对客观、冷酷、一成不变的校验器 (Validator)。比如，“MEX 必须是未出现的最小非负整数”、“字典序必须满足条件”。

战局被极度压缩了： 计算机根本不操心“我应该填什么”，它只负责无脑生成动作（遍历 0 到 m），然后把动作扔给校验器。战局（状态）仅仅变成了几个极简的变量（当前的 MEX 值、字典序 Flag）。

**优雅的本质： 规则永远站在那里不动，战局像流水一样经过。只要经过的时候符合规则，战局就继续往前流动；不符合，当场蒸发。这就实现了逻辑的完美解耦 (Decoupling)。**
